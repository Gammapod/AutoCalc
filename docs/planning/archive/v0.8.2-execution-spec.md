# v0.8.2 Execution + Expansion Spec

Status: Locked for implementation handoff (pre-work only).  
Target release: v0.8.2  
Source milestone: `design_refs/Milestones.md` (feature sets #2 and #3)

## 1. Behavior Contract

This release adds:

- `step_through` execution key with face `▷` (U+25BB).
- `[ ??? ]` settings toggle for per-step expansion rendering.

This release does not add new math operators or change non-step `=` behavior.

Required invariants:

- One `step_through` press executes exactly one operation slot.
- Partial step execution never appends to roll.
- Roll append happens exactly once when execution reaches terminal outcome (success or error).
- `=` during partial step executes only remaining slots.
- If `step_through` is not present on keypad, step state is inert and no step visuals are rendered.

## 2.3 New calculator execution state

Add to `CalculatorState`:

```ts
type StepProgressState = {
  active: boolean;
  seedTotal: CalculatorValue | null;
  currentTotal: CalculatorValue | null;
  nextSlotIndex: number;
  executedSlotResults: CalculatorValue[];
};
```

Semantics:

- `active=false` means no partial execution session.
- `seedTotal` is the session start total.
- `currentTotal` is the most recent step result (or seed for index `0`).
- `nextSlotIndex` is next operation slot to execute.
- `executedSlotResults[i]` is post-slot result for slot `i`.

Initialization:

- Initial/reset state: inactive, null totals, `nextSlotIndex=0`, empty results.

Persistence:

- Persist this state in normal save payload.
- Hydrate validation rule: if invalid shape or index bounds fail, coerce to inactive default.

## 3. Execution Semantics

## 3.1 `step_through` presence rule

`step_through` is considered present only if a keypad cell in `ui.keyLayout` contains key `exec_step_through`.  
Storage-only presence does not count.

If not present:

- Any existing step progress is cleared before processing execution actions.
- White-slot highlighting is disabled.
- `[ ??? ]` has no display effect.

## 3.2 Starting a step session

On `PRESS_KEY(exec_step_through)`:

1. Finalize drafting slot using existing finalize path.
2. If committed slot count is `0`, return unchanged state.
3. If no active session, initialize:
   - `active=true`
   - `seedTotal = calculator.total`
   - `currentTotal = calculator.total`
   - `nextSlotIndex = 0`
   - `executedSlotResults = []`

## 3.3 Per-press execution

For active session press:

1. Execute exactly slot at `nextSlotIndex` against `currentTotal`.
2. On success:
   - push result into `executedSlotResults`
   - set `currentTotal` to result
   - increment `nextSlotIndex`
3. If `nextSlotIndex` reaches slot count, finalize terminal success (section 3.5).
4. If slot execution errors, finalize terminal error immediately (section 3.5).

## 3.4 `=` behavior with active step session

On `PRESS_KEY(exec_equals)` while step is active:

- Execute remaining slots from `nextSlotIndex` to end using `currentTotal` as seed.
- Do not restart from slot `0`.
- Finalize once with one terminal roll append.

When step is inactive, keep existing equals behavior unchanged.

## 3.5 Terminal finalization

Terminal success/error (from `step_through` completion or `=` continuation) performs:

- `calculator.total = terminalResult`.
- Roll append exactly once (existing seed+step rules preserved).
- Overflow/error markers and unlock application use existing equals pipeline.
- Step progress reset to inactive default.

## 3.6 Step cancel/reset actions

Clear step progress to inactive default on:

- `RESET_RUN`
- `HYDRATE_SAVE`
- `UNLOCK_ALL`
- `PRESS_KEY` for `C`, `UNDO`, `←`
- `MOVE_LAYOUT_CELL`, `SWAP_LAYOUT_CELLS`
- `SET_KEYPAD_DIMENSIONS`, `UPGRADE_KEYPAD_ROW`, `UPGRADE_KEYPAD_COLUMN`

Rationale: these actions mutate function input state, history, or keypad topology.

## 4. UI + Rendering Contract

## 4.1 White next-slot highlight

Slot-token highlight only renders when all are true:

- `step_through` key is present on keypad.
- At least one committed operation slot exists.
- A valid step target exists.

Step target index:

- Active session: `nextSlotIndex`.
- Inactive session: `0`.

If index is out of range, render no highlight.

Rendering target:

- The operation-slot token in `[data-slot]` for the active step target is rendered white.
- Keypad keys are not highlighted for step targeting.

## 4.2 Inline executed-slot replacement

During active session, operation-slot display replaces each executed slot token with:

- `[ -> <displayValue> ]`

where `<displayValue>` is the calculator display string for `executedSlotResults[i]`.

Non-executed slots render with existing format.

## 4.3 `[ ??? ]` expansion rendering

`[ ??? ]` affects only the current step target token and only when:

- `step_through` is present on keypad.
- Step target index is valid.
- `STEP_EXPANSION_FLAG` is active.

Expansion source contract:

```ts
resolveStepExpansionText(slot: Slot, context: {
  seedTotal: CalculatorValue;
  currentTotal: CalculatorValue;
  nextSlotIndex: number;
}): string | null
```

Fallback:

- If resolver returns `null`, render normal token.
- Expansion mode never changes execution outcomes or state transitions.
