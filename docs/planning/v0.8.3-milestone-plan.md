# v0.8.3 Milestone Plan - Multi-Calculator Runtime

Date: 2026-03-15
Target release: v0.8.3

## Executive Summary

Current code quality is strong (build + test suite green), but architecture is only partially prepared for multi-calculator behavior.

- Readiness for "true per-calculator runtime state": medium-low.
- Recommended milestone strategy: 2 phases inside v0.8.3, with strict cut line after phase 1 if schedule slips.

## Readiness Estimate

### Overall readiness: 55%

What is ready:
- Test baseline is healthy: `npm test` passed all 88 test groups and UI complexity checks on 2026-03-15.
- Layout engine already has multi-instance concepts (`calculatorInstances`, `activeCalculatorId`, workbench order/offset).
- Domain code is modularized enough to migrate in slices (input/layout/lifecycle/flags split reducers).

What is not ready:
- Core domain state is singleton (`GameState` has one `calculator`, one `lambdaControl`, one `allocator`, one `ui`).
- Action/event model has no calculator scoping.
- Persistence schema and migration path are tightly coupled to singleton state.
- UI read models and module renderers read from singleton state directly.

## Key Evidence (Current Code)

- Singleton root state and action model:
  - `src/domain/types.ts` (`GameState`, `Action`)
  - `src/domain/state.ts` (`initialState`, `SAVE_SCHEMA_VERSION = 18`)
- Singleton reducer orchestration:
  - `src/domain/reducer.ts` (`reduceLegacy`, root `reducer`)
- Persistence tightly bound to singleton shape:
  - `src/infra/persistence/localStorageRepo.ts` (`toSerializableState`, `fromSerializableStateV3`)
  - `src/infra/persistence/migrations.ts` (`migrateToLatest`)
- UI + app layers assume one active calculator state object:
  - `src/app/autoEqualsScheduler.ts`
  - `src/app/ui/bootstrapUiController.ts`
  - `src/ui/shared/readModel.ts`
- Multi-instance support exists only in layout snapshot layer:
  - `src/ui/layout/types.ts`, `src/ui/layout/layoutEngine.ts`, `src/ui/layout/layoutAdapter.ts`
  - `tests/uiLayoutEngine.test.ts`

## Systems That Should Become Per-Calculator

Required for correctness with more than one calculator:
- Calculator execution state:
  - total, roll entries, roll analysis, operation slots, drafting slot, step progress.
- Control matrix / lambda control + allocator projection state.
- Calculator-local UI state:
  - keypad layout/cells, storage layout, active visualizer, memory variable, button flags.
- Calculator-local counters currently used by progression logic:
  - key press counts, allocator return/allocate press counts.

Recommended to keep global (for v0.8.3):
- Unlock definitions/catalog.
- Completed unlock ledger, unless product direction changes to per-calculator progression.

## Proposed Target Model (v0.8.3)

Introduce a multi-calculator root while preserving compatibility:

- `GameStateV2` (conceptual)
  - `activeCalculatorId: string`
  - `calculators: Record<string, CalculatorAggregateState>`
  - `order: string[]`
  - `sharedProgression: { unlocks, completedUnlockIds, ... }`

Where `CalculatorAggregateState` groups previous singleton fields now owned per instance:
- `calculator`
- `lambdaControl`
- `allocator`
- `ui`
- `keyPressCounts`
- `allocatorReturnPressCount`
- `allocatorAllocatePressCount`

## Milestone Scope Plan

## Phase 1 (Must Ship in v0.8.3): Multi-calculator foundation + one unlockable second calculator

Deliverables:
- New root state shape with active calculator routing.
- Action/event scoping with `calculatorId` (explicit or resolved via active id).
- Reducers migrated to update one calculator aggregate at a time.
- Persistence schema bump with migration from singleton saves to multi saves.
- UI renders active calculator from store; calculator switch action works.
- Auto-equals scheduler operates against active calculator instance only.
- Test coverage for:
  - routing correctness,
  - save/load migration,
  - active switch behavior,
  - parity invariants for active calculator paths.

Exit criteria:
- Existing single-calculator behavior preserved when only one calculator exists.
- Second calculator can be created/unlocked and switched to without state leakage.
- No cross-calculator contamination of roll, slots, or lambda/allocator values.

## Phase 2 (Should Fit in v0.8.3, but can cut if needed)

Deliverables:
- UX polish for multi-calculator workbench interactions.
- Additional per-calculator UI/runtime test depth (gesture + module-level edge cases).
- Cleanup of legacy aliases and temporary compatibility helpers introduced in phase 1.

Cut rule:
- If phase 1 is not code-complete and stable, defer phase 2 polish to v0.8.4.

## Rewrite Work Breakdown and Effort

Assumption: one experienced contributor, full-time equivalent, with current test harness retained.

1. Domain model and reducer scoping: 3-5 days
2. Action/event contract update (`calculatorId` routing): 2-3 days
3. Persistence schema + migration + back-compat tests: 3-5 days
4. UI read model + module integration for active calculator: 3-5 days
5. App runtime integration (scheduler, bootstrap controller, store wiring): 2-3 days
6. Tests + fixture updates + parity stabilization: 3-5 days
7. Buffer for regressions and migration edge cases: 2-4 days

Estimated total: 18-30 engineering days.
Practical release estimate: ~3-5 calendar weeks.

## Risk Register

1. Persistence migration risk (high)
- The migration file is large and currently assumes singleton payloads across many schema versions.
- Mitigation: add one dedicated migration stage (singleton -> multi root) and preserve old parser path for one release cycle.

2. Hidden singleton couplings in UI/read model (high)
- Many modules read `state.calculator` and related singleton paths directly.
- Mitigation: introduce selector helpers (`getActiveCalculatorState`) first, then migrate call sites incrementally.

3. Parity fixture churn (medium-high)
- Snapshot/parity fixtures are heavily coupled to serialized state shape.
- Mitigation: add compatibility serializer for fixtures during transition and update fixtures in a single controlled pass.

4. Scope creep into progression semantics (medium)
- Per-calculator progression vs shared progression can explode scope.
- Mitigation: freeze progression as global for v0.8.3 and explicitly defer per-calculator progression policy.

## Recommended v0.8.3 Acceptance Checklist

- Domain
  - Active calculator routing tests for every action family.
  - No leakage tests across calculator instances for roll/slots/lambda.
- Persistence
  - Old singleton save migrates to multi-root save deterministically.
  - Round-trip preserves active calculator and secondary calculator state.
- UI/App
  - Active calculator switch updates all dependent modules correctly.
  - Auto-equals and debug controls target active calculator only.
- Regression
  - Existing 0.8.2 behavior unchanged in one-calculator mode.

## Suggested Immediate Next Steps

1. Create `GameStateV2` and selector layer without changing behavior.
2. Migrate reducer/action routing to active-calculator scoping.
3. Add schema v19 migration singleton->multi and round-trip tests.
4. Wire UI/app to selectors, then update parity fixtures once.
